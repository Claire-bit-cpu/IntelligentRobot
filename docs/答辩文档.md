## 1. 项目概述

### 1.1 项目背景
本项目是一款基于飞书开放平台的智能机器人系统，旨在提升研发团队的协作效率。系统集成了 AI 智能理解、DevOps 工具链、代码审查、任务管理等多种功能，通过飞书机器人实现智能化的工作流自动化。

### 1.2 技术栈
- **后端框架**：Spring Boot 3.2.5
- **编程语言**：Java 17
- **AI 能力**：通义千问 API（DashScope SDK）
- **数据存储**：Redis（上下文管理、任务状态、缓存）、SQLite（FTS5 搜索引擎）
- **DevOps 集成**：GitHub API、GitLab API、JIRA API、Jenkins
- **监控**：Spring Boot Actuator、Prometheus、Grafana
- **构建工具**：Maven

### 1.3 核心功能模块
1. **AI 智能理解**：自然语言意图识别，支持多轮对话
2. **指令系统**：多级指令分发，支持权限控制、上下文感知
3. **DevOps 集成**：GitHub/GitLab CI/CD 管理、代码审查、部署触发
4. **通知系统**：智能降噪（去重 + 合并）
5. **任务管理**：异步任务状态跟踪、进度推送
6. **权限管理**：动态权限配置（Redis 热更新）

### 2.2 数据流设计

**Webhook 事件处理流程**：
1. 飞书平台发送事件到 `/feishu/webhook`
2. `WebhookController` 进行签名验证、解密、限流检查
3. 生成 `taskId`，创建任务状态记录
4. 提交到异步线程池处理
5. 立即返回 200 响应（避免飞书超时重试）
6. 异步处理完成后，通过飞书 API 发送回复

**消息处理流程**：
1. 解析消息内容（兼容标准结构和扁平结构）
2. 群聊检查是否 @机器人
3. 消息去重（内存 ConcurrentHashMap）
4. 分发到 `MessageDispatcher`
5. 根据四级响应策略处理
6. 发送回复（支持文本和卡片消息）

### 2.3 存储设计

**Redis 存储结构**：
```
# 对话历史

                                 # TTL: 5 分钟

# 用户上下文

                                 # TTL: 5 分钟（自动续期）
ctx:local:{userId}:{type}       # String 结构，局部上下文（JSON）
                                 # TTL: 由指令定义

# 任务状态

                                 # TTL: 1 小时

# 幂等性控制

# 动态配置

```

**SQLite 存储结构**：

## 3. 核心功能实现

### 3.1 四级响应策略（MessageDispatcher）

**设计思路**：根据用户消息的不同特征，采用不同的处理策略，优化用户体验。

**实现逻辑**：
```java
// Level 4: AI 智能理解（优先处理非/开头的消息）
if (!text.startsWith("/") && isBotMentioned) {
    String aiResult = aiUnderstandingService.processNaturalLanguage(...);
    if (aiResult != null) return aiResult;
}

// Level 3: 非@机器人的非指令消息，静默丢弃
if (!text.startsWith("/")) {
    return null;  // 不回复
}

// Level 1: 精确匹配
if (commandRegistry.hasCommand(commandName)) {
    return executeCommand(...);
}

// Level 2: 模糊匹配
List<MatchResult> suggestions = fuzzyMatcher.match(...);
if (!suggestions.isEmpty()) {
    return generateFuzzySuggestion(...);
}
```

**创新点**：
- 引入 AI 智能理解（Level 4），支持自然语言交互
- 模糊匹配使用 Levenshtein 距离算法，阈值可配置
- 无效消息静默丢弃，减少噪音

### 3.2 AI 智能理解（AiUnderstandingService）

**设计思路**：通过 LLM 进行意图识别，将自然语言转换为系统指令。

**Prompt 工程**：
```
你是一个智能助手，负责理解用户的自然语言消息，并识别其意图。

【意图类型】
1. command - 用户想执行某个指令
2. chat - 用户只是闲聊或提问
3. clarify - 信息不足，需要追问

【重要规则】
- 用户说"修改日程"、"改日程" → command: updateschedule
- 用户说"创建日程"、"新建日程" → command: schedule
- 绝对禁止将 command 类型识别为 chat 类型！
- 你只是意图识别器，不是执行器，不要编造执行结果！
```

**安全检测**：
- 检测 AI 生成的回复是否包含虚假执行结果
- 关键词：`已修改`、`已更新`、`已创建`、`修改成功` 等
- 如果发现虚假执行，拒绝回复并提示用户使用 `/` 指令

**对话历史管理**：
- 使用 Redis List 存储最近 50 条对话
- TTL: 5 分钟（用户连续对话时自动续期）
- 构建 AI 上下文时，自动加载最近 10 条历史

### 3.3 上下文感知（ContextManager + CommandRegistry）

**设计思路**：支持多轮对话，自动填充参数，提升用户体验。

**全局参数**：
- 作用域：所有支持上下文的指令
- 存储：Redis Hash（`ctx:global:{userId}`）
- TTL: 5 分钟（用户活跃时自动续期）
- 示例：`/weather 北京` 后，5 分钟内直接 `/weather` 可复用城市

**局部上下文**：
- 作用域：特定指令类型
- 存储：Redis String（`ctx:local:{userId}:{type}`）
- TTL: 由指令定义（默认 5 分钟）
- 示例：`/gitlog frontend 10` 后，直接 `/gitlog` 可复用参数

**参数自动填充优先级**：
1. 当前输入的参数（最高优先级）
2. 局部上下文参数
3. 全局参数（最低优先级）

### 3.4 异步任务状态跟踪（AsyncTaskStatus + TaskStatusService）

**设计思路**：长时间运行的任务需要进度反馈，避免用户焦虑。

**任务状态机**：
```
PENDING → PROCESSING → COMPLETED
                  └→ FAILED
```

**进度推送**：
- 任务开始：推送 `⏳ 任务处理中... (0%)`
- 任务进行中：更新进度 `⏳ 任务处理中... (50%)`
- 任务完成：更新 `✅ 任务完成 (100%)`
- 任务失败：更新 `❌ 任务失败`

**飞书消息更新优化**：
- 使用 `updateMessage` API 原地更新消息（非发送新消息）
- 频率控制：同一任务最多每 2 秒更新一次（避免限流）
- 降级策略：更新失败时，自动发送新消息

**慢任务通知**：
- 阈值：3 秒（`task.slow-threshold-ms`）
- 超过阈值的任务，完成时推送通知到群聊
- 快速任务不推送通知，避免刷屏

### 3.5 智能降噪通知（NotificationService）

**设计思路**：DevOps 工具会产生大量通知，需要去重和合并。

**去重逻辑**：
- 基于内容哈希去重
- 时间窗口：5 分钟（`notification.dedup.window-seconds`）
- 示例：同一 PR 的多次 push 通知，只推送一次

**合并逻辑**：
- 时间窗口：5 分钟（`notification.batch.window-seconds`）
- 阈值：5 条（`notification.batch.threshold`）
- 达到阈值后，推送合并摘要
- 示例：5 分钟内 10 个 PR 通知 → 合并为 1 条摘要

**通知优先级**：
- 普通通知：进入合并队列
- 紧急通知：跳过合并队列，立即推送（如部署失败）

### 3.6 权限管理（AuthService）

**设计思路**：支持动态配置权限名单，无需重启服务。

**权限级别**：
```
NONE < DEVELOPER < ADMIN
```

**动态配置**：
- 存储：Redis（`config:dynamic:auth:{type}`）
- 热更新：修改后立即生效
- 降级策略：Redis 不可用时，fallback 到静态配置（`application.yaml`）

**管理接口**：
```
GET  /api/auth/users          # 查看权限名单
POST /api/auth/users          # 添加用户到权限组
DELETE /api/auth/users/{openId}  # 移除用户权限
```

### 3.7 代码审查（CodeReviewService）

**设计思路**：利用 AI 自动审查代码质量，提升代码评审效率。

**审查维度**：
1. 代码质量评分（1-10 分）
2. 安全问题检测
3. 性能问题分析
4. 修改建议

**自动触发**：
- Push 事件：自动审查最新提交
- PR 事件：自动审查 PR 代码
- 可配置开关：`github.webhook.auto-review`

**审查结果推送**：
- 格式化后的审查报告
- 推送到飞书群聊
- 包含评分、问题列表、建议等

### 3.8 GitHub Webhook 集成（GitHubWebhookService）

**设计思路**：实时同步 GitHub 事件到飞书群聊，提升团队协作效率。

**支持的事件**：
1. `push`：代码推送通知
2. `pull_request`：PR 状态变更通知
3. `pull_request_review`：PR 审查通知

**去重逻辑**：
- 系统创建的分支：跳过通知（避免噪音）
- 纯分支创建（无提交）：跳过通知
- `synchronize` 事件：跳过通知（与 push 重复）

**自动代码审查**：
- PR 创建时：自动触发审查
- Push 时：自动审查最新提交
- 审查结果推送到飞书群聊

## 4. 技术难点与解决方案

### 4.1 飞书消息结构兼容性

**问题**：飞书 Webhook 回调的消息结构有两种格式：
1. 标准结构：`{"header": {...}, "event": {...}}`
2. 扁平结构（解密后）：`{"event_id": "...", "event_type": "...", "event": {...}}`

**解决方案**：
- 优先从 `header` 中提取 `event_id` 和 `event_type`
- 如果失败，直接从 body 中提取
- 使用 `ObjectMapper.convertValue()` 进行类型安全的转换

```java
// 兼容两种结构
FeishuCallback callback = objectMapper.convertValue(body, FeishuCallback.class);
if (callback.getEvent() != null && callback.getEvent().getMessage() != null) {
    chatId = callback.getEvent().getMessage().getChatId();
} else {
    // 从 eventMap 直接解析
    chatId = (String) eventMap.get("chat_id");
}
```

### 4.2 异步处理时的任务状态跟踪

**问题**：Webhook 需要快速返回响应（<10ms），但业务逻辑可能耗时较长。如何在异步处理后跟踪任务状态？

**解决方案**：
1. 生成 `taskId`（UUID），创建任务状态记录
2. 提交到异步线程池处理
3. 立即返回 200 响应（包含 `task_id`）
4. 异步处理完成后，通过飞书 API 发送回复
5. 支持查询任务状态：`GET /api/task/{taskId}`

**任务状态存储**：
- 内存模式：`ConcurrentHashMap`（适用于单实例）
- Redis 模式：`String` 结构（适用于集群部署）

### 4.3 消息去重与幂等性

**问题**：飞书平台会在超时后重试 Webhook 请求，导致重复处理。

**解决方案**：
1. **幂等性控制**：
   - 使用 `event_id` 作为唯一标识
   - Redis 存储已处理的 `event_id`（TTL: 24 小时）
   - 重复事件直接返回成功

2. **消息去重**：
   - 使用 `message_id` 作为唯一标识
   - 内存 `ConcurrentHashMap` 存储已处理的 `message_id`
   - 时间窗口：30 秒（`dedup.window-ms`）
   - 定时清理：每 10 分钟清理过期记录

### 4.4 AI 意图识别的准确性

**问题**：LLM 可能误判用户意图，导致错误执行指令或返回无意义回复。

**解决方案**：
1. **Prompt 工程**：
   - 明确指令列表和对应关系
   - 添加自我检查规则（Q1-Q4）
   - 强制要求返回 JSON 格式

2. **安全检测**：
   - 检测 AI 生成的回复是否包含虚假执行结果
   - 如果发现虚假执行，拒绝回复并提示用户使用 `/` 指令

3. **降级策略**：
   - 解析失败时，降级为 `chat` 意图
   - LLM 调用失败时，返回友好提示

### 4.5 上下文管理的内存泄漏

**问题**：用户在多轮对话后，上下文数据可能长久占用内存。

**解决方案**：
1. **TTL 机制**：
   - 全局参数：TTL 5 分钟（用户活跃时自动续期）
   - 局部上下文：TTL 由指令定义（默认 5 分钟）
   - 对话历史：TTL 5 分钟

2. **自动清理**：
   - 切换到新指令时，清空之前的局部上下文
   - 强独立性指令（如 `/jira 1234`）不继承上下文

3. **Redis 持久化**：
   - 上下文数据存储到 Redis
   - 避免应用重启后丢失上下文
   - 支持集群部署

### 4.6 飞书 API 限流

**问题**：飞书 API 有严格的限流策略（如消息发送 QPS 限制）。

**解决方案**：
1. **消息合并**：
   - 将多条通知合并为一条摘要
   - 减少 API 调用次数

2. **频率控制**：
   - 同一任务最多每 2 秒更新一次飞书消息
   - 使用 `ConcurrentHashMap` 记录上次更新时间

3. **降级策略**：
   - 更新消息失败时，自动发送新消息
   - API 限流时，记录日志并等待重试
